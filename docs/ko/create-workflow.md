# /create — 아이디에이션에서 검증까지

> 원시 아이디어에서 C-레벨 리뷰를 거친 검증된 제품 컨셉까지 — 하나의 명령어로.

[English →](../en/create-workflow.md)

---

## 언제 사용하나요

| 상황 | 명령어 |
|------|--------|
| **기획이 전혀 없음** — 처음부터 시작 | `/create "내 아이디어"` |
| **대규모 신규 기능** — 투자 전 검증 필요 | `/create "결제 기능 추가"` |
| **이미 spec이 있음** — 빠른 디자인 탐색 | `/brainstorm "기능"` |
| **spec 존재** — /dev 전 제품 리뷰 게이트 | `/office-hours 기능이름` |

`/create`는 **가장 초기 단계** — `/spec` 이전, `/dev` 이전. 검증된 컨셉을 얻은 후 spec으로 변환하여 개발합니다.

---

## 파이프라인: 5단계

```
Phase 1: 컨텍스트     프로젝트 상태 읽기 (기존 프로젝트인 경우)
    ↓
Phase 2: 질문         6개 핵심 질문 (한 번에 하나, 사용자 언어로)
    ↓
Phase 3: 접근법       2-3가지 대안 + 트레이드오프 → 사용자 선택
    ↓
Phase 4: C-레벨 리뷰  CEO → CTO → CPO → CMO 순차 리뷰
    ↓
Phase 5: 결정         VISION + REVIEW + DECISION 문서 → 선택적 spec 변환
```

### Phase 1: 컨텍스트 수집

- `spec/PROJECT.md`와 `spec/ARCHITECTURE.md` 읽기 (있는 경우)
- `spec/feature/` 스캔하여 기존 피처 파악
- 설명에서 사용자 언어 감지

### Phase 2: 핵심 질문

수요 현실과 제약사항을 드러내는 6가지 질문. **한 번에 하나씩** 사용자 언어로:

1. **수요 현실** — 이 문제를 가진 사람은 누구? 지금 어떻게 해결하나요?
2. **전제 도전** — 아무것도 안 하면 어떻게 되나요?
3. **타겟 유저** — 구체적으로 누구? 그들의 워크플로우는?
4. **성공 기준** — 성공하면 어떤 지표가 움직이나요?
5. **제약사항** — 바꿀 수 없는 것은?
6. **10점 만점 경험** — 완벽한 경험은 어떤 모습?

이미 답한 질문은 건너뜁니다. "그냥 해줘"라고 하면 빠르게 진행됩니다.

### Phase 3: 접근법 생성

항상 2-3가지 대안 생성:

- **Minimal Viable** — 가장 빠른 출시, 최소 스코프
- **Ideal Architecture** — 장기적 최선의 설계
- **Creative/Lateral** — 예상 밖의 접근 (선택)

각각: 노력도, 위험도, 장점, 단점, "적합한 상황" 포함.

### Phase 4: C-Suite 팀 리뷰

5명의 전문가가 **병렬로** 평가한 뒤, 이견이 있으면 토론합니다:

| 역할 | 초점 | 스킬 |
|------|------|------|
| **CEO** | 비전, 스코프, 수요, 리소스 배분 | investor-materials, investor-outreach |
| **CTO** | 아키텍처, 실현 가능성, 기술부채, 보안 | architectures, vercel-react-best-practices |
| **CPO** | 유저 가치, 스토리, 지표, UX | pm-product-strategy, brainstorming |
| **CMO** | 포지셔닝, 경쟁, 메시징, 성장 | marketing-psychology, copywriting |
| **CDO** | 디자인, 정보 아키텍처, 접근성 | frontend-design, brainstorming |

**Round 1**: 5명 모두 독립 평가 (병렬). 각각 **APPROVE**, **CONCERN**, **BLOCK** 반환.
**Round 2** (조건부): BLOCK 또는 상충 의견 → 관련 에이전트에게 상대 의견 공유 → 재평가.

### Phase 5: 결정 & Spec 변환

리뷰 후 `spec/create/[이름]/`에 세 가지 문서 생성:
- `VISION.md` — 문제, 수요, 타겟 유저, 가치 제안
- `C-REVIEW.md` — 5개 C-suite 평가 + 토론 요약
- `DECISION.md` — 선택한 접근법, 핵심 결정, 제약사항

선택: **spec으로 변환할까요?**
- **예** → 복사 가능한 `/spec` 명령어 제공
- **아니오** → 문서는 나중 참고용으로 저장

## 산출물 문서

모든 문서는 `spec/create/[이름]/`에 저장 — `spec/feature/`와 완전히 분리됩니다.

| 문서 | 최대 줄수 | 내용 |
|------|----------|------|
| VISION.md | 80 | 문제, 수요, 타겟 유저, 가치 제안, 성공 기준 |
| C-REVIEW.md | 100 | CEO/CTO/CPO/CMO/CDO 평가 + 판결 + 위험 |
| DECISION.md | 60 | 선택한 접근법, 핵심 결정, 제약사항, spec 변환 상태 |

---

## 토큰 격리

`/create` 문서는 다른 명령어에서 **절대 로드되지 않습니다**:
- `/spec`은 `spec/feature/[이름]/`만 읽음
- `/dev`는 `spec/feature/[이름]/`만 읽음
- `/init`은 `spec/` 최상위만 스캔

일반 개발 워크플로우에 **토큰 비용 제로**.

---

## Spec 변환

변환 선택 시 `/create`가 출력:

```
/spec [이름] "[문제 + 접근법 + 제약사항 요약]"
```

Claude Code에 붙여넣기. spec-writer가 요약을 기반으로 빌드 — `spec/create/` 파일을 자동으로 읽지 않습니다.

경계 유지: `/create`는 아이디에이션, `/spec`은 명세.

## 예시

### 새 프로젝트:
```
/create "냉장고 사진으로 레시피를 추천하는 AI 앱"
```

### 기존 프로젝트의 새 기능:
```
/create "문서 에디터에 실시간 협업 추가"
```

### 검증 후:
```
/spec recipe-app "냉장고 재고 기반 AI 레시피 추천. 최소 접근: 이미지 인식 + 레시피 DB 조회. 제약: 오프라인 동작 필수."
```

## FAQ

**Q: /create 전에 /init을 실행해야 하나요?**
아닙니다. greenfield 프로젝트에서도 동작합니다. `spec/PROJECT.md`가 있으면 컨텍스트로 활용하고, 없으면 그냥 진행합니다.

**Q: C-레벨 리뷰를 건너뛸 수 있나요?**
현재는 불가 — 파이프라인의 일부입니다. 각 리뷰는 최대 15줄로 간결합니다.

**Q: C-레벨 리뷰어가 BLOCK하면?**
우려 사항이 하이라이트됩니다. 대응하거나, 접근법을 수정하거나, 그대로 진행할 수 있습니다. BLOCK은 권고이지 게이트가 아닙니다.

**Q: /create가 /spec이나 /dev에 영향을 주나요?**
아닙니다. 토큰 격리로 영향 제로. `/create`는 `spec/create/`에 쓰며, 다른 명령어는 이를 읽지 않습니다.

**Q: 같은 컨셉에 /create를 다시 실행할 수 있나요?**
예. 기존 `spec/create/[이름]/` 문서를 덮어씁니다.
